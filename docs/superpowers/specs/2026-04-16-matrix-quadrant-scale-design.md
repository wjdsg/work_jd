# Matrix Quadrant Layout & 10-Level Scale Design

## 1. 背景与目标
- 调整矩阵象限位置：
  - 左上：重要不紧急
  - 右上：重要且紧急
  - 左下：不重要不紧急
  - 右下：不重要但紧急
- 将重要性与紧急性评分从 1-5 扩展为 1-10。
- 对历史数据执行线性迁移：`new = old * 2`。
- 保持现有中文 UI 语言与视觉风格不变，仅修正行为语义与交互范围。

## 2. 约束与非目标
- 不改动核心模块边界（features/store/services/storage 目录结构保持不变）。
- 不引入新的三方库。
- 不改动当前提醒、备份、遥测的业务语义。

## 3. 方案选择
### 方案对比
1. 显式迁移（推荐）
   - 在 IndexedDB schema migration 中完成 1-5 到 1-10 的一次性转换。
   - 优点：数据模型统一、后续逻辑简单。
   - 缺点：需补 migration 测试与边界处理。
2. 运行时映射
   - 存储仍为 1-5，UI 临时映射为 1-10。
   - 优点：改动小。
   - 缺点：长期技术债高，排序/过滤/导入导出容易不一致。
3. 仅新数据升级
   - 历史数据不迁移，新数据 1-10。
   - 优点：最快。
   - 缺点：同库双标尺，体验与统计不一致。

### 决策
- 采用方案 1（显式迁移）。

## 4. 设计细节
### 4.1 数据模型与校验
- 将 `importance`/`urgency` 的合法范围由 `1..5` 调整为 `1..10`。
- 所有输入校验、默认值、边界 clamp 逻辑同步升级。

### 4.2 Migration 设计
- 新增 schema version（例如 v2）。
- v1 -> v2 升级流程：
  1. 遍历 tasks，将 `importance` 与 `urgency` 执行 `value * 2`。
  2. 对结果做 `clamp(1, 10)`。
  3. 同步迁移 settings 阈值（`importance` 与 `urgency` 同样 `*2` 并 clamp）。
  4. 触发/执行象限重算，保证 `quadrant` 与阈值一致。
- 幂等性：migration 仅在版本升级路径执行一次，不重复放大。

### 4.3 象限映射与布局
- 坐标语义固定：
  - X 轴（左 -> 右）：紧急性低 -> 高
  - Y 轴（下 -> 上）：重要性低 -> 高
- 象限标题与映射关系显式定义，避免通过视觉顺序隐式推断。
- 调整 Matrix 渲染顺序/网格定位，确保右上为“重要且紧急”。
- 边界判定规则（必须统一）：
  - `importance >= threshold.importance` 视为“高重要”。
  - `urgency >= threshold.urgency` 视为“高紧急”。
  - 象限归属：
    - `high importance && high urgency` => q1（右上，重要且紧急）
    - `high importance && low urgency` => q2（左上，重要不紧急）
    - `low importance && high urgency` => q3（右下，不重要但紧急）
    - `low importance && low urgency` => q4（左下，不重要不紧急）

### 4.4 交互与 UI
- TaskForm 与编辑面板 slider 改为 `min=1, max=10, step=1`。
- 详情面板、统计文案与筛选逻辑使用新范围。
- 保持现有中文文案风格，仅更新必要标签。

### 4.5 迁移异常值策略与责任点
- 迁移执行点：仅在 IndexedDB `upgrade(v1 -> v2)` 流程中同步执行并写回，禁止在渲染层或读取层做懒迁移。
- 迁移顺序：
  1. 读取旧值。
  2. 旧值标准化（见下条）。
  3. 执行 `normalized * 2` 并 `clamp(1, 10)`。
  4. 迁移阈值后重算 `quadrant` 并写回。
- 旧值标准化规则：
  - `null/undefined/NaN/Infinity/-Infinity`：回退到默认值 `3`（旧量程中值）。
  - 非整数：先四舍五入到最近整数，再进入后续流程。
  - 越界值：先在旧量程 `1..5` 内 clamp，再做 `*2`。
- 迁移失败策略：
  - 单次 upgrade 事务内任一写入失败则回滚事务，不写入部分迁移结果。
  - 应用侧记录错误日志并保留旧版本数据，等待下次启动重试。

### 4.6 迁移对照样例
| 场景 | 旧值 | 标准化后 | 新值（*2 + clamp） |
| --- | --- | --- | --- |
| 正常边界下限 | 1 | 1 | 2 |
| 正常边界上限 | 5 | 5 | 10 |
| 中间值 | 3 | 3 | 6 |
| 非整数 | 2.4 | 2 | 4 |
| 非整数 | 2.6 | 3 | 6 |
| 空值 | null | 3 | 6 |
| 非数值 | NaN | 3 | 6 |
| 越界低值 | 0 | 1 | 2 |
| 越界高值 | 7 | 5 | 10 |

## 5. 测试策略（TDD）
先写失败测试，再改实现：
1. 矩阵映射测试
   - 验证四象限标题对应位置，尤其右上=重要且紧急、左上=重要不紧急。
   - 验证阈值边界归属：当 `importance === threshold` 或 `urgency === threshold` 时按“高”处理（即使用 `>=`）。
2. 迁移测试
   - 给定 v1 数据（1-5），迁移后应为 `*2`（并正确 clamp）。
   - 阈值迁移后与象限重算一致。
   - 给定异常旧值（null/NaN/非整数/越界）时，迁移结果符合“标准化规则”。
   - 幂等性：同一数据在已升级版本再次打开，不发生二次放大。
3. 表单与模型校验测试
   - 允许 1 与 10，拒绝越界输入。
   - 检查所有滑杆 `min=1,max=10,step=1`（新建、编辑、设置阈值）。
4. 回归验证
   - `npm run lint`
   - `npm run test -- --run`
   - `npm run build`

## 6. 风险与缓解
- 风险：旧数据迁移后任务分布变化（阈值也放大时总体分区语义保持）。
- 风险：遗漏某些校验点仍按 1-5。
- 缓解：通过全局搜索范围常量与失败测试覆盖关键路径。

## 8. 验收标准
- 象限布局与命名符合目标位置。
- 新建/编辑任务可选择 1-10。
- 历史 1-5 数据升级后为 `*2`，无重复迁移。
- 全量 lint/test/build 通过。

### 8.1 Given/When/Then 验收条目
- Given 旧库中任务为 `importance=5, urgency=3` 且阈值 `3/3`，When 执行 v1->v2 升级，Then 任务分值为 `10/6`，阈值为 `6/6`，并按新阈值重算象限。
- Given 旧库存在 `importance=null`，When 执行升级，Then 该值按默认 `3` 迁移为 `6`。
- Given 应用已在 v2 运行一次，When 再次重启，Then 已迁移任务分值保持不变，不会从 `10` 变成 `20`。
- Given 用户在表单/详情/设置调节滑杆，When 选择最小和最大值，Then 可稳定写入并显示 `1` 与 `10`。
- Given 任务值等于阈值边界，When 渲染矩阵，Then 按“高”侧归类（`>=` 语义）。
